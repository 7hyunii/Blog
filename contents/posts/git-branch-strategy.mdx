팀 프로젝트에서 효과적으로 사용할 수 있는 Git 브랜치 전략을 비교합니다.

## Git Flow

가장 널리 알려진 브랜치 전략입니다.

### 브랜치 구조

- **master**: 프로덕션 배포용
- **develop**: 개발 브랜치
- **feature**: 기능 개발
- **release**: 배포 준비
- **hotfix**: 긴급 수정

### 장단점

**장점**
- 명확한 역할 분리
- 대규모 프로젝트에 적합

**단점**
- 복잡한 브랜치 구조
- CI/CD와 맞지 않을 수 있음

## GitHub Flow

더 단순한 접근 방식입니다.

### 특징

1. master 브랜치는 항상 배포 가능한 상태
2. 새로운 작업은 feature 브랜치에서
3. Pull Request로 코드 리뷰
4. 리뷰 후 master에 머지하고 즉시 배포

### 적합한 프로젝트

- 지속적 배포를 하는 프로젝트
- 작은 팀
- 빠른 개발 사이클

---

프로젝트의 특성과 팀의 배포 주기에 맞는 전략을 선택하는 것이 중요합니다.
